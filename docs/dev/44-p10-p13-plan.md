## 开发顺序

1. 先把 P10 的真实样本语料、质量标签和聚合指标落稳。
2. 再补 P11 的浏览器级 E2E，把页面主链路锁住。
3. 然后推进 P12，先锁 provider 契约测试，再补 request/response 正规化。
4. 再做 P13，默认 dry-run，严格限制清理范围，并把 SQLite 归为人工备份策略。
5. 每完成一个大块就回归，最后做全量回归与真实样本烟测。

## 验收门槛

- `py -m pytest` 全绿。
- `py -m app.tools.input_runner --path input\软著材料 --mode single_case_package` 维持 `unknown=0`。
- `py -m app.tools.input_runner --path input\合作协议 --mode batch_same_material` 维持稳定结果。
- `py -m app.tools.runtime_cleanup` 默认 dry-run，不会误删 active log / SQLite。

## 风险与对策

- 风险：自定义 pytest runner 会跑全量而不是子集。
  - 对策：把“局部测试”也当成全量回归来设计，新增测试避免依赖高级 pytest 特性。
- 风险：Windows 下 SQLite 文件可能被别的进程占用。
  - 对策：`clear_database()` 先尝试删文件，失败则回退到表级清空。
- 风险：运行时清理脚本容易误删调查数据。
  - 对策：只自动清理 `submissions / uploads / logs`，SQLite 只给备份建议，不自动删库。
