# P3-P4 经验手册

## 下载链路经验

- 本地兼容层和真实框架的差异，最容易在下载响应这类“边界对象”上暴露，而不是在业务逻辑里暴露。
- 下载接口测试不能只断言状态码，还要同时覆盖：
  - `Content-Disposition`
  - 响应内容是否非空
  - 二进制 bundle 是否可解压

## 浏览器操作台经验

- 只做 API 不够，人工纠偏场景必须在 Submission 页面上形成可点击、可回跳、可追溯的闭环。
- 每增加一个 operator action，最好同步补：
  - 一个 HTML action 测试
  - 一个页面显隐断言
  - 一个导出或日志断言

## AI 边界经验

- 隐私要求不能只写在文档里，必须拆成独立代码层：
  - 本地脱敏
  - AI safe payload 构建
  - payload 安全校验
  - provider 解析与 fallback
  - provider adapter
- 默认 provider 要保持 `mock`，这样开发和回归不会被外部依赖绑死。
- 非 mock provider 最好先接一个本地 `safe_stub`，先验证“边界”正确，再接真实网络调用。

## 回归策略经验

- 新增一批浏览器端能力后，先跑最贴近该链路的集成测试，再跑全量回归，定位速度最快。
- 每轮回归后都要用真实样本补一轮烟测，否则很容易出现“测试通过但真实材料退化”。
- 对隐私边界改动，除了默认模式，还要加一条“启用非 mock provider”的真实烟测。
