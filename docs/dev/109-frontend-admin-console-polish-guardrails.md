# Frontend Admin Console Polish Guardrails

## Why This Round Worked Better

- 先确认产品类型，再做界面。
- 先修编译与结构问题，再做视觉增强。
- 复用现有后台设计系统，而不是重新发明一套页面语言。
- 用页面契约测试兜底，避免“看起来变好了，实际上破了”。

## New Lessons Added

### Lesson 1

- 前端“不好看”不一定是颜色和排版问题。
- 在这个项目里，更常见的问题是：
  - 页面结构没有对齐现有 CSS 语义
  - 导致浏览器里看起来像裸文本或奇怪的大按钮

### Lesson 2

- 管理系统的核心不是 hero、banner、氛围图，而是：
  - KPI
  - 表格
  - 状态
  - 操作入口
  - 报告阅读
  - 运维可观测性

### Lesson 3

- 每次前端调整都必须同时验证三层：
  - 源码可编译
  - 页面契约测试通过
  - 关键页面结构仍保留既定名词与入口

## Mandatory Checks For Future UI Rounds

1. 先判断这是后台系统、分析系统、阅读器，还是展示页。
2. 如果是后台系统，首屏必须优先展示：
   - 导入入口
   - 状态摘要
   - 数据表格
   - 操作入口
3. 优先检查页面 class 是否真的命中了现有 CSS 体系。
4. 每次重构后先跑 `py_compile`，再跑页面契约测试。
5. 若页面涉及下载、报告、运维入口，必须检查它们不是退化成普通裸链接。

## Anti-Patterns

- 不要为了“更高级”而把后台做成 landing page。
- 不要抛弃已存在的样式系统再从零造轮子。
- 不要只检查首页；Case、Report、Ops 也要同步统一。
- 不要只看 HTML 生成是否成功，还要看编译与路由契约是否正常。

## Reusable Rule

- 在这个项目里，前端最优策略通常是：
  - 先对齐信息架构
  - 再对齐 class 语义
  - 最后补视觉 polish
