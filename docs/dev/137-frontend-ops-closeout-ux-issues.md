# 运维页交付收尾可视化问题记录
## 日期

- 2026-04-21

## 发现的问题

### 1. 业务收尾能力存在，但没有进入主视图

- 现象：
  - `delivery_closeout` 已经能输出结论，但用户只能通过命令行或直接翻 `docs/dev` 查看。
- 风险：
  - 运维和业务仍会优先盯探针或闸门，而忽略“是否已经可交付”这个更高层结论。
- 处理：
  - 在 `/ops` 顶部区域直接加入业务收尾 KPI、状态卡和主面板。

### 2. 直接在页面里动态重跑 closeout 成本不合适

- 现象：
  - 如果前端每次打开都重跑 closeout，会让页面读取和产物生成耦合。
- 风险：
  - 页面访问变重，状态刷新也不稳定。
- 处理：
  - 新增只读状态加载与下载 helper，页面只消费 latest 产物。

### 3. `/ops` 新增第五个 KPI 后桌面端会掉行

- 现象：
  - 原来的 KPI 栅格是四列，新增“业务收尾”后会出现一张卡掉到第二行。
- 风险：
  - 页面顶部第一屏的秩序感被破坏。
- 处理：
  - 新增 `kpi-grid-ops`，为运维页单独使用五列栅格。

### 4. 下载 helper 需要显式阻止路径逃逸

- 现象：
  - closeout 历史产物路径如果不做约束，理论上会存在路径拼接风险。
- 风险：
  - 下载路由可能被拼出非预期文件。
- 处理：
  - 对 `file_name` 做了扁平文件名校验。
  - 增加契约测试覆盖嵌套路径拒绝行为。

## 当前残余风险

- `/ops` 现在已经能看见 closeout，但真实截图级视觉确认仍需要浏览器人工复看。
- 如果后续 closeout checks 继续扩容，主面板中的检查表可能需要进一步分组。
