# P0 问题与处理

## Issue 1

- 现象：真实 ZIP 中的中文文件名被错误解码，导致分类器无法命中文件名关键词。
- 影响：`2502`、`2505` 中多个材料被错误落入 `unknown`。
- 根因：归档成员名在读取时存在 `cp437` 默认解码与真实编码不一致的问题。

## Fix 1

- 策略：对 ZIP 成员名尝试 `cp437 -> utf-8` 与 `cp437 -> gb18030` 恢复，并按可读性评分选优。
- 结果：`2505` 的 2 个 unknown 全部消失；`2502` 的 3 个 unknown 降至 1 个。

## Issue 2

- 现象：二进制 `.doc` 的噪声文本会触发内容规则，存在误判风险。
- 影响：系统会把低质量文本当成可自动分类内容，结果不可解释也不稳定。
- 根因：原有管线缺少 parse quality 概念，分类器只知道“命中规则”，不知道“文本是否可信”。

## Fix 2

- 策略：加入 parse quality 评分、质量等级、质量原因、可用性判断，并在低质量文本上阻断内容型自动分类。
- 结果：低质量二进制文本会被保留为 `unknown`，同时给出明确原因，例如：
  - `binary_doc_parse_failed`
  - `blocked_low_quality_content_signal`

## Issue 3

- 现象：即使分类结果正确，运营侧也无法一眼识别哪些材料需要人工复核。
- 影响：P1 人工纠错无法顺畅开展。
- 根因：Submission 页面缺少 review queue 视图。

## Fix 3

- 策略：在 Submission 详情页增加 `Needs Review` 队列，并展示：
  - 当前材料类型
  - 解析质量
  - 复核原因
  - 状态
- 结果：待复核材料已可视化，后续可直接承接人工纠错工作流。

## 当前残留风险

- `.doc` 的真实文本提取仍偏弱，尤其是老式二进制 Word 文档。
- 这不是 P0 的漏做，而是当前解析器能力边界。
- 系统现在已经能把这类问题显式标为 low quality，并推入待复核队列。
